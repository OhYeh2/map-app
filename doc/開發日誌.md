# 荒野相簿地圖 - 開發與除錯日誌

## 發生時間
2026年5月9日 (優化)

## 功能更新：大量原圖載入記憶體優化 (OOM 防護)
### 問題描述
當使用者匯入包含 200 張以上的高解析度照片時，由於原本的程式碼採用 `URL.createObjectURL(file)` 直接將千萬畫素的原始圖片傳遞給地圖做為縮圖標記使用。此舉迫使瀏覽器在背景將所有原圖進行解碼，導致記憶體（RAM / GPU）瞬間被佔用數 GB，極易造成網頁卡頓甚至因記憶體耗盡 (Out of Memory, OOM) 而崩潰。

### 解決方案
實作了雙重影像壓縮與快取優化機制 (`src/utils/mediaParser.ts`)：
1. **內建微型縮圖抽取 (優先)**：透過 `exifr` 套件，直接讀取照片標頭 (EXIF) 內建的微型縮圖 (Micro-thumbnail)。此方法完全不需將原圖載入記憶體解碼，速度達到毫秒級。
2. **Canvas 背景動態壓縮 (備案)**：若遇到無內建縮圖的相片（或 PNG 檔），系統會在背景利用 HTML5 `<canvas>` 將原圖強制限縮至 256x256 以下的低解析度尺寸，產生輕量級的 JPEG Blob 後，立刻釋放原圖記憶體 (`URL.revokeObjectURL`)。

### 結果
在地圖上同時顯示數百張圖片，記憶體使用量從原本的數 GB 驟降至幾十 MB 等級，大幅提升了大規模檔案載入時的穩定性與拖曳操作的流暢度。

# 荒野相簿地圖 - 開發與除錯日誌

## 發生時間
2026年5月9日

## 問題描述：滑鼠懸停數字引發的黑畫面崩潰
使用者回報一項錯誤：當將滑鼠游標移至地圖上的群組數字（例如標示有 19 張照片的叢集）時，群組雖然似乎有展開，但隨即**整個網頁畫面變黑**，且在開發者工具 (F12) 中看不到任何明顯的 React 錯誤訊息。

經過分析錯誤畫面（網址變更為 `chrome-error://chromewebdata/`），確認該「黑畫面」並非 UI 樣式遮擋，而是 **Google Chrome 瀏覽器分頁發生崩潰 (Aw, Snap!)**。

## 根本原因分析 (Root Cause)
這個崩潰的底層原因是一個高頻率的 **React 無限渲染與事件觸發迴圈 (Infinite Render/Event Loop)**。具體過程如下：

1. **觸發展開**：滑鼠懸停於群組數字時，觸發了 `onMouseOver` 事件，呼叫 `spiderfy()` 將叢集展開，同時將該叢集寫入 React 的 State (`setSpiderfiedCluster(cluster)`)。
2. **重新渲染**：State 的更新導致 `Map.tsx` 元件觸發重新渲染 (Re-render)。
3. **子元件重建**：在渲染過程中，地圖標記的陣列 `{items.map(...)}` 產生了新的 React Elements，且傳入的事件處理函式（如 `onMarkerOut`）也因為 inline function 的寫法而獲得了新的參考記憶體位址 (Reference)。
4. **叢集套件誤判**：`react-leaflet-cluster` 偵測到其子節點 (Markers) 的 Props 發生變化，判定標記已更新。為了確保一致性，套件會**清除所有標記並重新加入**。
5. **強制收合 (Unspiderfy)**：清除標記的動作會強制解除原本已展開的狀態 (Unspiderfy)，導致展開的縮圖瞬間消失，而帶有數字的群組圖示重新回到地圖上的原始座標。
6. **滑鼠重新觸發**：因為使用者的滑鼠**依然停留在原來的座標位置**，剛被重新渲染出來的群組圖示立刻又與滑鼠發生碰撞，再次觸發 `onMouseOver`。
7. **死亡迴圈**：系統又回到步驟 1，於一秒內循環數千次，耗盡瀏覽器記憶體與 CPU 資源，最終導致分頁崩潰黑屏。

## 解決方案
為了徹底打破這個迴圈並提升地圖效能，我們進行了以下架構重構：

1. **移除觸發渲染的 State，改用 Ref 追蹤狀態**：
   將 `const [spiderfiedCluster, setSpiderfiedCluster] = useState(null)` 替換為 `const spiderfiedClusterRef = useRef(null)`。
   * **效果**：當叢集展開並紀錄當下狀態時，不會觸發 React 的重新渲染，直接從源頭阻斷迴圈。

2. **事件函式快取 (Callback Refs)**：
   利用 `useRef` 建立 `callbacksRef` 來儲存最新的事件處理常式（例如 `onMarkerHover`、`onMarkerOut` 等）。
   * **效果**：保證在標記物件上的 Event Handlers 永遠拿到最新的函式，而不需要依賴 React 重新賦予新的屬性。

3. **地圖標記的快取 (useMemo)**：
   將整個 `<MarkerClusterGroup>` 包裝在 `React.useMemo` 內，將相依陣列限定為 `[items, markerSize]`。
   * **效果**：只有在「真實檔案資料」或「縮圖大小拉桿」發生改變時，地圖標記才會被重新建立。滑鼠的移動與 Hover 將不再對 DOM 產生破壞性的重建。

4. **解耦滑鼠追蹤器 (MapEventsHandler)**：
   讓負責判斷「滑鼠是否離開同心圓範圍」的追蹤器元件，直接讀取 `spiderfiedClusterRef`，實現自動收合 (Auto-Collapse) 功能而無須依賴 State 驅動。

## 測試結果
修正後，將滑鼠移至大量的照片群組上，能平順地展開成放射狀的多層同心圓（半徑依數量動態計算）。即使在展開的縮圖間移動游標以觸發「右下角影片/照片預覽」，也不會發生任何卡頓或黑畫面，解決了崩潰問題。
